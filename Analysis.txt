Discussion:

We see that the latency for the US server is lower than the EU server because the Firestore datastore is in US-central. So when the EU server tries to write to or read the database, there is a longer network path and the data has to travel a physically longer distance. For the US server on the other hand, there is a shorter distance and shorter network path.

We also see that the latency for register is a lot higher than list. This is because of durability guarantees that the database has to make for write operations. The database must ensure that data is actually present, whereas reads just fetch what is already there in the database (not making any changes, so need to make any guarantees before completing the read operation). There are also other complexities in write operations like replication (our Firestore datastore is sharded) and updating indices in the datastore, that read operations do not need to deal with.

Although, Firestore with datastore mode is eventually consistent, the results from my experiment showed that there were no instances of a username not being found after registration. This result was initially surprising, given that I ensured that the two servers were truly in different regions (as seen in the latency differences) and that I immediately made the get request after the post request. I even used Python sessions to prevent overhead on the networking side. But the fact that I wasn't able to see the stale data means that the duration in which the request was stale for was smaller than the time it took for the connection TO DO
